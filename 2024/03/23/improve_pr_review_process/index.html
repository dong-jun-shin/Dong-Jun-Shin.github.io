    코드 리뷰 문화를 개선하고 생산성을 올려보자

    배경

    제가 속해있는 제품 서비스 팀은 5명으로 구성되어 2주 스프린트 단위로 개발을 진행하는 조직이었습니다. 저희 팀은 배포할 시기가 되면 기존 계획했던 일정이 부족하여 배포 예정에 포함되어 있던 이슈를 다음 스프린트로 옮기는 일이 잦았었는데, 이는 외부 일정의 차질과 협업 관계에서 신뢰를 떨어뜨리는 심각한 문제라고 생각하여 분석을 해보았습니다.

    일정이 지연되는 이유 중 여러가지가 있겠지만 외부보다는 팀 내부에서 개선할 부분이 있을지 살펴보았고, 주로 PR 리뷰에서 예측했던 일정이 지연되는 것을 확인하였습니다. 이번 글은 일정 지연 빈도를 줄이기 위해 PR 리뷰 프로세스의 개선을 시도한 내용과 문제를 해결하기 위해 리서치했던 과정을 정리한 내용을 담아 보았습니다.

    현재 프로세스

    현재 1개의 이슈를 처리할 때, 다음과 같은 단계를 거치게 됩니다.

    1. 작업
2. PR 생성
3. 리뷰 요청
4. 대기
5. 리뷰 반영
6. 4 ~ 5를 반복한 후 승인
7. master로 merge

    문제점

    주로 3번까지는 담당자들이 열심히 집중하여 빠르게 올라왔습니다. 하지만 4번에서 리뷰어들이 바빠 리뷰의 우선순위가 밀리면서 1차 지연이 발생했고, 리뷰를 반영 후 다시 리뷰를 요청하면서 n번에 거듭한 지연이 발생하고 있었습니다.

    PR 리뷰 과정에서 여러 문제가 생기고, 이로 인해 생산성이 저하되는 것을 확인했으니 더 정확한 원인을 파악해볼 차례입니다. 저는 PR 리뷰를 할 때 병목이 생기는 부분을 명확하게 구분해보고자, 리뷰를 하는 사람과 리뷰를 받는 사람의 관점으로 나누어 보았습니다.

    1. 리뷰를 하는 사람

    먼저 리뷰를 하는 사람 입장에서 가용 리소스, 코드 리뷰 관점으로 나누어 볼 수 있습니다.

    • 가용 리소스 관점의 문제
      • 자신이 맡은 이슈를 포함하여 PR 리뷰보다 우선 순위가 높은 일들이 많았습니다.
        • PR 리뷰도 충분히 중요한 일이라는 점은 이미 모두가 인지하였지만, 내 업무를 하기에도 시간이 부족한 상황입니다.
      • MSA 구조에서 PR리뷰를 하면 PR이 여러개 서비스 repo에 섞여있어, 리뷰 대상 PR이 명확하지 않은 문제가 있었습니다.
        • 리뷰가 필요한 PR, 작업중인 PR, (Sub PR을 합칠 목적의)메인 PR 등이 섞여있어 리뷰 대상을 파악하기가 힘든 상황입니다.
      • PR이 일정 마감 직전에 리뷰되는 문제가 있었습니다.
        • PR 리뷰를 하기 부담스러운 문제로 인해 미루어지고, 일정이 촉박할 때 리뷰가 되면서 문제를 놓치는 경우가 많아지게 됩니다.
    • 코드 리뷰 관점의 문제
      • main 브랜치에 merge하기 위해서는 팀장의 승인이 필요하여, 한 명에게 책임과 권한이 집중되면서 병목이 발생하고 있었습니다.
        • 병목이 심화되면서, 리뷰는 점점 더뎌지고 리뷰를 하는 사람, 받는 사람 모두 지치는 상황입니다.
      • PR 템플릿과 티켓의 내용이 정리되어 있지 않아, 작업 담당자 이외의 사람이 파악하는데 지연되는 문제가 있었습니다.
        • 해당 PR에 대한 내용을 쉽게 파악할 수 있도록 간결하고 명확한 정리가 중요
      • 코드에서 의미를 유추, 해석, 의도의 파악이 어려울 때 지연
        • 이미 서로 lint에 명시된 컨벤션 정도는 맞추었지만, 리뷰에서 이해에 도움을 주기 위해 컨벤션을 좀 더 다듬으면 좋을 것 같다고 생각

    2. 리뷰를 받는 사람(작업 담당자)

    그리고 리뷰를 받는 사람 입장에서는 업무 진행 관점으로 볼 수 있습니다.

    • 업무 진행 관점의 문제
      • 리뷰가 오래 걸릴수록, 동시에 진행하는 업무로 인한 생산성 하락 문제가 있었습니다.
        • 동시에 관리하는 업무가 많은 상황에서 생산성은 떨어지고 일정 마감일이 되어 리뷰가 완료되면 이는 생산성을 더 떨어뜨리게 됩니다.
      • 다른 업무를 진행할 때 필요한 스위칭 비용이 높아지는 문제가 있었습니다.
        • 현재 업무를 정리해두고, 다른 업무 시작을 위해 파악하는 시간이 필요한 상황입니다.
      • 여러 작업을 동시에 파악하고 관리하고 있어야 하기 때문에 더 많은 리소스를 사용해야 하는 문제가 있었습니다.
        • 현재 진행 상황을 체크하고 일정을 준수할 수 있도록 여러가지 상황을 고려하는 작업이 많아지게 되는 상황입니다.

    문제 현황 분석

    앞서 살펴본 문제들로 인해 지연이 있다고 판단하였고, 어느 정도의 병목이 있는지 파악해보고자 했습니다.

    • PR 리뷰에 소요된 시간 
      전체 PR 리뷰 기간: 09/05 ~ (다음 해)03/06
총 PR 개수: 468개
평균 PR 리뷰 완료 시간: 3일 7시간 22분 21초
최소 PR 리뷰 완료 시간: 12초
최대 PR 리뷰 완료 시간: 1개월 17일 3시간 16분 38초

    우선 저는 저희 팀이 맡고 있는 메인 서비스 저장소를 선택하여 반년동안 진행했던 PR 리뷰에 대해 분석을 해보았습니다. 그 결과, 평균적으로 리뷰를 진행하기 위해 PR 건당 3 ~ 4일이 소요되고 있었고, 길게는 1개월하고도 17일이 걸리기도 하였습니다.

    간단한 Key 교체, 환경설정 변경, 배포하기 위해 급하게 승인됐던 PR들을 고려하면, 1 ~ 2주 정도의 체감 소요 시간이 얼추 맞았습니다.

    목표 수립

    문제를 해결하기 전, 우리가 기대한 결과가 어떤 모습일지 명확하게 해야 합니다. 결과를 명확하게 함으로써, 방향을 명확하게 하고 우리가 문제를 해결하는데 집중할 수 있도록 합니다. 그럼 앞서 나열한 문제점이 해결되었을 때, 우리가 기대하는 결과는 어떤 모습일까요?

    제가 생각해본 결과는 다음과 같았습니다.

    • 개인이 리뷰 시간을 분리해서 할당하고, 업무 관리를 위해 진행 상황 공유를 하도록 업무 프로세스 개선
    • 명확하게 구분할 수 있도록 리뷰 대상 PR을 리스트업
    • 일정 마감 직전에 문제가 되지 않도록 리뷰 대상 PR에 자체 우선순위를 부여
    • 리뷰 승인을 부담없이 할 수 있도록 여러 명에게 책임을 나누어 병목 최소화
    • 내용을 쉽게 파악할 수 있도록 해당 PR에 대해 간결하고 명확한 일감 정리
    • 이미 서로 lint에 명시된 컨벤션 정도는 맞추었지만, 리뷰에서 빠른 코드 이해를 위해 기존 컨벤션 정책 개선

    목표 달성 시 기대효과

    그리고 세운 목표를 달성했을 때, 어떤 변화가 있을지도 생각해봅니다.

    • PR 리뷰를 Spot성 업무가 아닌 계획 가능한 업무로 만듦
    • 컨벤션과 그라운드 룰 수립, 명확하고 간결한 내용(이슈, 코드) 전달을 통해 리뷰어의 가독성, 생산성을 향상
    • 이슈가 오래 방치되지 않도록 하여, 동시에 진행하는 업무 개수, 비용을 감소

    이제 문제를 해결해보자

    문제를 파악하고 목표를 수립하였으며 목표 달성에 따른 기대효과까지 정리를 하였습니다. 이제는 해결하는 일만 남았습니다.

    이번 개선 과정은 구성원들에게 큰 부담을 지우지 않도록 하여 개선된 프로세스가 잘 정착되는 것을 우선으로 몇가지 개선 방안을 생각해보았습니다. 기존에 하던 방식이 아니기 때문에 개인의 부담이 아예 없을 수는 없지만, 전체적으로는 모두의 부담을 덜고 생산성이 높아질 수 있을 것이라고 생각했습니다.

    해결 방안 요약

    1. Issue managing Template 개선
    2. 팀 리뷰 문화 구축 및 정착
    3. PR Notification System 도입

    Issue managing Template 개선

    1. PR Template

    • 개발자만 보는 내용으로 개발자에게 의도를 전달하는 목적으로 작성
    • 템플릿은 이슈 설명, 해결을 위한 접근 방법, 추가적으로 변경된 내용, 신경써서 보기 원하는 부분, 리뷰 시 참고 사항으로 5가지 항목으로 구성하였습니다. 이 5가지는 처음 작업을 파악하기 위해, 궁금해 할 것이 무엇일 지, 리뷰를 하는 사람 입장에서 생각하여 구성한 항목입니다.
    [이슈 설명]
> 어떤 기능을 구현한건지, 이슈 대응이라면 어떤 이슈인지 PR이 열린 배경과 목적을 리뷰어들이 쉽게 이해할 수 있도록 적어주세요. (잘 정리된 티켓 링크로 대체 가능)
- {수정하고자 하는 상황}

[해결을 위한 접근 방법]
> 문제를 해결하면서 주요 변경된 부분들에 대해 적어주세요.
- {수정한 방법}

[추가적으로 변경된 내용]
> 리팩토링, lint 적용 등 이슈 해결 외에 변경된 사항을 적어주세요. (없을 경우, "없음")
- 없음

[신경써서 보기를 원하는 부분]
> 개발 과정에서 다른 분들의 의견이 궁금했거나 꼭 확인해주었으면 하는 부분
- {요청하고 싶은 내용}

[리뷰 시 참고 사항]
- {리뷰어가 알아야 할 작업 사항}
- {리뷰어가 알아야 할 이슈 사항}

    정리한 템플릿은 PR에 적용하면 다음과 같이 사용할 수 있습니다.

    [PR에 실제 템플릿을 적용한 예시]

    2. Jira 이슈

    • 개발자, 비개발자 모두가 보는 내용으로 비개발적인 내용을 전달하는 목적으로 작성
    • Jira Template App을 활용하는 방법으로 적용 가능
    • 템플릿은 As-Is, To-Be, Reference로 정리하여, 비개발적인 관점으로 구성하였습니다.
    [As-Is]
- {현재 상황}

[To-Be]
- {작업 후 상황}

[References]
- {히스토리}
- {참고 링크}
- {기타. 이슈와 관련된 사항}

    팀 리뷰 문화 구축 및 정착

    각자의 스타일로 작업을 하다 보니, 한눈에 읽히기 어려운 점이 있었습니다. 코드 리뷰는 협업을 하는 것이기 때문에, 컨벤션을 정해서 리뷰 지연을 해소해보고자 하였습니다.

    먼저 팀 단위의 리뷰 문화를 구축하기 위해 논의가 필요했고, 다 같이 논의할 수 있도록 업무 프로세스, PR 리뷰, Code 관점에서 그라운드 룰과 컨벤션에 대한 초안을 정리해보았습니다.

    • 업무 프로세스
    - PR 리뷰도 하나의 업무입니다.
- 다른 사람의 PR을 리뷰하는 것을 자신의 일정 관리에 고려하지 않아서 지연이 되는 경우가 빈번했습니다.
- PR 리뷰도 하나의 업무로 우선순위와 함께 일정에 고려헤서 팀 단위에서 집중할 수 있는 시간을 확보하는 것이 중요합니다.
    • PR 리뷰
    - 변화를 작게 유지합니다.
  - 수정사항이 많아지면 리뷰어가 파악하는데 필요한 시간과 에너지는 더 커집니다.
  - PR을 올릴 때 300~500줄 목표로 시작하고, 팀에 맞는 컨벤션으로 개선을 합니다.
- 리뷰는 자주, 짧은 세션으로 진행합니다.
  - 리뷰어의 부담을 덜고, 계획적으로 업무하기 좋은 여건이 됩니다.
- 의미있는 PR을 만들기에 충분한 정보를 제공합니다.
  - 리뷰어가 파악하는데 필요한 시간을 많이 아낄 수 있습니다.
  - 템플릿 등을 함께 활용한다면 가독성을 더욱 높일 수 있습니다.
- Approve는 팀 내 누구나 할 수 있습니다.
  - Approve는 책임보다는 의견을 표현하는 것입니다.
  - 리뷰는 코드의 완벽함이 아닌 최선을 위해 하는 것입니다.
- Label 지정하기
  - 마감 기한 설정, 배포 유형, QC 필요 여부 등 업무 진행에 변화를 줄 수 있는 부분을 표시합니다.
  - PR을 구조화함으로 파악하는 시간을 줄이고, 자동화를 적용하며 협업에 필요한 과정을 줄일 수 있습니다.
    • Code convention
    - 주석, draft PR, PR 템플릿, PR 코멘트 등을 활용합니다.
  - 설명이 필요한 부분을 표시함으로써 협업 과정의 불필요한 소통을 줄일 수 있습니다.
  - 설명이 필요한 부분은 작업이 추가적으로 필요한 부분, 코드만으로 이해하기 어려운 부분, 특이사항이 있는 부분 등이 있습니다.

    이렇게 초안을 작성하였습니다. 작성한 초안은 팀에 맞는 문화를 구축하기 위해 제안을 하였고, 문화가 정착될 수 있도록 지속적인 추가/수정 피드백 요청과 리뷰 문화가 팀에 맞도록 개선 요청을 제안하였습니다.

    PR Notification System

    이제 정립한 그라운드 룰과 컨벤션을 팀에서 잘 지켜질 수 있도록 하는 일이 필요합니다. 수립한 정책이 지켜지고 있는지 확인하며 요청하거나 PR을 관리하는 과정에서 직접 PR 마감 기한을 변경하며 관리해야 한다면, 구성원들에게 개선보다 부담으로 다가올 것입니다.

    그래서 부담이 될 수 있는 부분을 대신 담당해줄 시스템의 도입을 생각하였고, PR Label과 리마인더 봇을 만들게 되었습니다. 우선 시스템이 어디까지 담당할 지 역할의 범위를 정하는 것이 필요합니다.

    저는 앞서 말했던 부담이 될 수 있는 부분을 시스템의 역할로 보고, 다음과 같이 정의하였습니다.

    • 리뷰해야 하는 PR에 지정된 label을 기준으로 우선순위를 정리하는 역할을 수행합니다.
    • 우선순위, 보류된 PR, 초과된 PR등 리뷰해야 하는 PR을 모아서 주기적으로 알려주는 역할을 수행합니다.
    • PR에 지정된 마감 기한 Label의 관리와 필수 Label, 타이틀, 본문 등의 부재를 확인하고 요청하는 역할을 수행합니다.

    PR Label 활용

    먼저 시스템에서 활용할 수 있도록 마감 기한, PR 유형, 진행 상태 등을 Label 유형으로 정의하였습니다.

    • 마감 기한의 경우, PR당 3일 내에 다루어 질 수 있도록 D-n, 기한 초과로 구성하였습니다.
    • PR 유형은 업무 진행 유형을 반영하여 신규기능, 기능개선, 오류수정, 기타수정과 QC 여부를 다룰 수 있도록 하였습니다.
    • 진행 상태는 merge 대기, 보류를 구분하였고 별도의 진행 상태가 없이 기한이 있다면 진행 중인 상태로 판단하였습니다.

    PR 리마인더 봇

    이제 PR Label을 도입하면서 시스템을 만들기 위한 준비가 끝났습니다. 남은 일은 PR Label을 이용해서 부담이 될 부분을 담당해줄 봇을 만들어야 합니다.

    봇의 이름은 Coral-bob이라고 정의하였는데, Coral-bob이 역할을 잘 수행할 수 있게, 다음의 기능을 하도록 구현되었습니다.

    - PR이 Open된 상태이고 MergeReady가 아닌 상태를 대상으로 관리를 합니다.
- 대상 PR의 상태에 따라 label을 확인하고 변경합니다.
  - 날짜가 지남에 따라 D-day 라벨을 관리합니다.
    - D-3 ~ D-1일 사이면, 하루 줄어든 라벨로 변경합니다.
    - D-0일인 경우, 기한이 초과되었다고 판단하고 OverDay 라벨로 변경합니다.
  - MergeReady
    - Approve를 받았다면 D-n 라벨을 제거하고 MergeReady 추가합니다.
  - 배포 유형, QC 여부 라벨 체크
    - 필수 라벨이 없는 경우, 추가해달라는 코멘트를 PR에 추가합니다.
- 알림 발송
  - Block 분류, D-day별, OverDay로 분류하여 정리합니다.
  - PR당 `PR 제목 + D-day`로 구성하며, Jira 이슈를 포함한 경우 URL이 첨부되도록 합니다.
  - 리뷰할 PR이 없는 경우, 팀을 격려하는 메시지를 대신 발송합니다.
- 배포 내역 정리
  - 배포 준비를 할 때, Type label, QC 여부 label을 이용해서 배포 내역 초안을 작성해줍니다.

    마치며

    이렇게 Issue managing Template과 팀 리뷰 문화를 위한 그라운드 룰과 컨벤션 정립, PR Notification System 도입을 해보았습니다.

    앞으로 새로운 팀원의 원활한 합류와 팀의 높은 생산성이 더욱 절실해질 상황에서 기존 시스템의 높은 복잡도, 기술 문서의 부재, 그라운드 룰과 팀 문화의 부재 등 해결해야 할 문제는 큰 부담으로 느껴지고 있습니다.

    한 번에 모든 문제를 해결하면 좋겠지만, 많은 문제들에 매몰되지 않도록 꾸준히 시도하고 해결해나가는 것이 이럴 때일수록 중요할 것입니다. 지속가능한 팀 문화의 한 부분으로 자리 잡을 수 있게, 계속 가설을 세우며 문제를 해결하는 문화가 정착하길 바라며, 그 과정의 하나로 시도했던 코드 리뷰 문화 개선에 대한 글을 마칩니다.
