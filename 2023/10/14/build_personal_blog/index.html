    배경

    처음에는 블로그 포스팅에 집중하고자 티스토리를 이용해서 개인 블로그를 시작했었는데, 글을 작성하다 보니 블로그 테마와 로깅 관리, 컨텐츠 관리, 블로그 이전 등 제약을 느끼게 되었습니다. 이러한 이유로 Github에 직접 호스팅해서 관리할 수 있는 개인 블로그로 옮기게 되었는데, 이 과정에서 마크다운 포스트를 작성하는 방법이나 포스트를 관리하는 방법이 궁금해서 편하게 할 수 있는 방법이 있는지 찾아보았었습니다.

    하지만 Github Pages를 이용한 블로그 관련 포스트 중 나에게 fit한 방법이 없어, 이번 기회에 나한테 fit한 환경을 하나씩 구성하면서 과정을 정리해보고자 합니다. 또 이 글을 통해 블로그를 구축하려는 누군가가 참고할 수 있도록 블로그를 가꾸어 나간 과정을 간략하게 소개하겠습니다.

    Github Pages를 선택한 이유

    Github Pages는 아래와 같은 장단점이 존재합니다.

    장점

    1. 블로그 레이아웃, 스타일부터 콘텐츠 방식까지 모두 자유롭게 커스텀이 가능하다.
    2. 블로그의 포스팅 이력뿐만 아니라, 블로그 관리를 위해 수정한 이력까지 모두 관리가 가능하다.
    3. 마크다운 형식으로 블로그 콘텐츠를 관리할 수 있어, 다른 블로그로 이전하거나 크롤링, 백업 등 활용하기 편한 이점이 있다.

    단점

    1. 블로그를 하나부터 열까지 직접 다 제작해야 한다.(테마나 템플릿을 이용하더라도 나한테 맞추려면 결국 다 파악하고 수정해야 한다…)
    2. 포스트 에디터부터 포스팅까지 모두 직접 수행해야 한다.
    3. 콘텐츠를 포함한 블로그 소스코드를 public으로 공개해야 한다.(private으로 운영하려면 유료 계정인 Github pro 계정만 가능하다.)

    위와 같은 장단점을 가지고 다른 플랫폼과 비교하여 장단점을 따져보았을 때, Github pages의 장점을 다른 서비스에서 얻기는 힘들지만 다른 서비스의 장점(Github Pages의 단점)은 Github Pages에서 충분히 보완 가능하다고 생각하여 이전할 블로그 서비스로 Github Pages를 선택하게 되었습니다.

    진행 과정

    1. 블로그 생성

    가장 먼저 내가 사용할 블로그를 만들어야 합니다. 블로그 생성의 시작으로 저는 사용할 템플릿을 고르는 작업을 하였습니다. 이전 티스토리의 경우, 다크 그레이 배경에 그린 포인트 테마를 사용했었는데 전체적으로 어두운 느낌이 드는게 싫었습니다. 그래서 이번에는 깔끔하면서 밝은 느낌을 줄 수 있는 테마를 찾았고, flexible-jekyll을 선택하게 되었습니다.

    [템플릿과 현재 블로그]

    템플릿을 선택했다면 Profile, Social link와 같은 블로그 정보를 내 정보에 맞게 수정하고 필요한 기능을 붙여야 합니다. 저는 video embed를 위해 youtube embed 플러그인도 추가하였습니다. 이 플러그인은 jekyll이 빌드될 때, html 문법에 맞게 변환해주는 플러그인으로, 아래와 같이 최종 산출물인 html에 동영상을 embed 해줍니다.

    [플러그인으로 추가한 유튜브 태그]

    블로그에 필요한 기능을 어느 정도 추가했다면, 이제 관리와 운영을 위한 작업도 해줍니다. 먼저 블로그 관리를 위해 방문자에 대한 정보를 파악할 수 있는 구글 애널리틱스 서비스와 검색과 블로그의 인덱싱 정보를 활용하기 위해 구글 서치 콘솔 서비스를 추가하였습니다. 그리고 SEO를 위해 구글 서치 콘솔에 등록할 robots.txt와 sitemap.xml를 추가하여 검색 엔진에 최적화하고 검색 결과에 나타날 수 있도록 하였습니다.

    마지막으로 Github Pro 계정이 아닐 경우, Pages repo는 public으로만 설정이 가능하여 블로그 관련 소스 코드가 공개되어 버리고 맙니다. 저는 이러한 유출을 막고자 Pages repo는 빌드된 결과물(HTML 파일 등)만 관리하고, 별도의 private repo에 소스 코드만 관리하는 방식으로 repo를 분리하였습니다. 이렇게 소스 코드와 배포 파일을 분리하게 되면서 포스팅할 때마다 여러번의 커밋을 해야 하는 문제가 생기는데, 이 문제는 아래의 배포 스크립트를 통해 자동화하는 방법으로 해결하였습니다.

    여기까지 과정을 통해, Github Pages를 사용해도 소스 코드의 보안을 유지하면서 나에게 맞는 fit한 블로그를 생성하는 단계가 마무리되었습니다.

    2. 콘텐츠 이전

    이제 다음 작업은 생성한 블로그에 이전 포스트를 옮기는 작업이었습니다. 이 단계에서는 티스토리가 제공하는 마이그레이션 기능을 이용하였습니다. 이 기능은 티스토리에 있는 블로그 포스트를 html, image 형태로 백업하는 기능인데, 블로그의 데이터를 쉽게 가져올 수 있어 크롤링할 수고를 덜었습니다.

    [티스토리 export 기능]

    백업 데이터를 바로 블로그에 쓸 수 있으면 좋겠지만, 아쉽게도 개인 블로그 구조에 맞는 형태로 변환이 필요합니다. 이 과정을 자동화하기 위해 티스토리의 블로그 백업 파일을 블로그에 구조에 맞게 import하는 html to markdown 변환 스크립트와 블로그 구조에 맞게 재구성하는 스크립트를 작성하였습니다.

    [블로그 포스트 구조]

    마지막으로 2개 스크립트를 돌려 블로그 구조에 맞는 포스트 폴더를 생성하고, 이 폴더를 포스트 폴더로 옮겨 블로그 이전을 완료했습니다.

    3. 포스팅 관리 및 퍼블리싱 자동화

    과거의 포스트까지 이전을 완료하고, 필요한 작업은 ‘앞으로 작성할 포스팅의 관리와 퍼블리싱 과정 자동화’였습니다. 포스트는 한 번에 작성하는게 아니기 때문에 포스트 소스를 모아놓는 폴더와 작성중인 포스트를 모아놓는 폴더가 필요했습니다. 그래서 블로그 소스 코드와 함께 관리할 수 있도록 repo에 ‘_drafts’ 폴더를 추가하고, jekyll로 build할 때 제외되도록 exclude에 추가하였습니다. 또 draft 폴더에서 관리되는 글들 중 작성이 완료된 포스트를 간편하게 퍼블리싱(블로그 구조에 맞게 위치)할 수 있도록 퍼블리싱 스크립트까지 작성하여 작성할 포스트, 작성중인 포스트, 작성한 포스트를 관리할 수 있는 환경을 만들었습니다.

    [스크립트를 포함한 ‘_drafts’ 폴더 구조]

    또 포스트에 썸네일, 작성 날짜, image 사용 시 명시하는 path와 같은 반복적으로 작성하는 단순한 내용을 간소화할 수 있도록 아래 내용을 퍼블리싱 스크립트에 추가하였습니다.

    1. 기본 사진이 없을 경우, 카테고리 대표 사진을 사용하도록 설정합니다.
    2. 작성한 날짜를 퍼블리싱하는 시점의 date를 사용하도록 설정합니다.
    3. image 위치를 명시하기 위한 공통 path를 변환합니다.

    마지막으로 블로그에 반영하는 배포 스크립트까지 작성하여, 2개의 스크립트(퍼블리싱, 배포)로 포스팅 과정을 자동화하였습니다.

    4. 포스트 생성을 위한 마크다운 에디터 환경 구축

    마크다운 작성을 하게 되면서 가장 먼저 찾은 것은 에디터였습니다. 간단한 문법은 습관적으로 쓸 수 있었지만, post layout, image, video 등 사용 빈도가 상대적으로 떨어지는 문법들은 매일 작성하지 않으면 금방 잊어 버리게 되어, 블로그 작성을 어렵게 만들기 때문이었습니다.

    이미 만들어진 에디터들이 이런 문제를 해결해주면 좋았겠지만, 아쉽게도 허용하는 문법이 조금씩 다르거나 작성한 마크다운이 블로그에서는 다르게 보이는 문제가 있었습니다. 그래서 저는 방법을 찾던 중 알게 된 vscode에서 제공하는 마크다운 미리보기 기능과 유용한 extension을 사용하여 이 문제를 해결하였습니다.

    먼저 vscode의 마크다운 미리보기는 css를 적용하는 것이 가능했는데, 블로그에 사용하고 있는 scss를 css로 변환한다면 이를 활용할 수 있었습니다. 그래서 scss를 하나의 폴더로 모은 후 모두 합친 css로 변환하고, vscode의 settings에서 markdown preview에 css 적용을 시도하였습니다. 다행이도 결과는 성공적이었고, 작성한 마크다운이 블로그에서 동일하게 보이는 환경을 구축할 수 있었습니다.

    [vscode에 블로그 css를 적용한 미리보기]

    그리고 문법을 기억하지 못하더라도 마크다운의 일관성을 유지할 수 있도록, 블로그와 동일한 문법을 제안해주는 snippet 기능과 작성 규칙을 정의할 수 있는 lint 기능을 vscode에 추가하여 문제를 해결하였습니다.

    snippet 기능의 경우, post layout, image, link 등 상대적으로 사용 빈도가 낮은 문법들을 추가해두어, 문법의 일관성과 포스트 작성의 편의성을 챙겼습니다.

    [마크다운 문법 스니펫 기능]

    또 markdownlint의 Rules를 참고하여, 포스트마다 규칙적인 구조로 작성할 수 있는 환경을 만들어 관리의 용이성을 높였습니다.

    [마크다운 lint를 적용한 에디터]

    5. 가독성 개선을 위한 블로그의 전체적인 스타일 개선

    이제 블로그 생성, 운영 및 관리, 포스트 관리, 포스팅까지 완료하니, 한글이 고려되지 않은 영문 템플릿을 사용함으로 생기는 문제와 포스트에 보이는 스타일이 맞지 않아 생기는 가독성 문제가 눈에 띄었습니다. 그래서 이 단계에서는 주로 스타일을 수정하였고, 자잘한 부분이 많아 말보다는 기록하는 형태로 정리를 하였습니다.

    • 각 scss 수정
      • 공통
        • 원하는 font를 추가하고 통일
        • font 굵기를 여러 종류 사용할 수 있도록 수정
        • 기본 font 굵기 설정
        • 글자간 spacing 초기화
        • identity color 수정
      • heading
        • 기본 굵기, 기본 spacing 변경
      • content
        • padding 수정
      • html tags
        • header, footer
          • margin, padding, border 수정
        • h
          • weight, margin 수정
        • table
          • 크기, 컬러, 선 등 css 전반을 수정
        • code
          • font 굵기, font 크기, 색상 수정
          • Syntax Highlighting 수정
        • block quote
          • color, font size 수정
        • hr
          • height, margin 수정
        • img
          • image, caption 에 대한 모양과 font size, padding, 정렬 수정
        • pre
          • padding 수정 ```

    마무리

    마무리로, 블로그를 만들며 필요한 환경이나 설정을 하나씩 덧붙여가면서, 역시 무엇 하나 쉽게 되는 것은 없다고 생각했습니다. 마음만은 하루 딱 잡아 끝내버리고 싶은데, 이것 역시 필요한 부분과 개선이 필요한 부분들은 계속해서 보이게 되는 것 같습니다. 큰 작업으로 카테고리를 나누어 보았을 때 이번이 블로그에 3번째 작업인데, 이쯤 되니 한 번에 만드는 것보다 MVP로 진행하고 점진적으로 개선하는 게 더 효율적인 거 같다는 생각을 다시금 하게 되었습니다.

    처음에는 배우거나 학습한 내용을 로컬에 메모장으로 간단하게 정리했다가, 블로그로 정리하는 것의 이점을 느끼게 되어 티스토리를 활용했었고, 제약을 느껴 이제 개인 블로그 구축까지 오게 되었습니다. 매번 필요를 느낄 때 개선하다 보니 과정마다 블로그 구축 및 관리에 대한 경험을 쌓을 수 있어 좋았고, 블로그를 통해 내 머릿속에 있는 파편화된 지식을 정리하는 것이 효과적인 학습 방법이라는 것을 느낄 수 있어 좋았던 것 같습니다. (블로그가 왜 좋은지도 머릿속에 있는 내용들을 정리해보고 싶은데, 별도 포스트로 써봐도 좋을 거 같습니다)

    이제 더욱 생산성이 올라간 환경에서 절약한 시간으로 기록하는 데 집중할 수 있게 되었습니다.
