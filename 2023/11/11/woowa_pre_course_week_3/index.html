    • 테스트를 작성해야 하는 이유를 생각하기
      • 테스트의 유용함을 언제 느꼈는가?
        • 장애가 발생하는 상황에서 제품의 고도화를 할 때 느낌
          • 서비스 운영 시, 서비스가 커져감에 따라 점점 장애가 많이 발생하는 것을 봄
          • 장애의 발생은 곧 팀의 운영에 제동이 걸리는 것을 의미했음
          • 이렇게 장애가 빈번하게 생길수록 신규 서비스의 개발에 지연을 발생시키고 제품의 신뢰는 하락하며, 구성원이 신규 기능을 만들거나 유지보수하는데 큰 두려움을 줌
        • 장애가 빈번하게 발생해서 악화된 상황이 오기 전에 무엇을 했다면 이런 상황을 피할 수 있었을까?
          • 테스트를 잘 작성했다면 이런 충격을 많이 완화할 수 있지 않았을까 싶음
          • 고도화할 때마다 테스트할 양은 많아지고 매번 테스트하기에 점점 많은 비용이 들게 됨
          • 고도화할 때마다 개발하는 시간 중 1/10만 투자해서 테스트 코드를 작성한다면 기능 개발이 끝날 때마다 같은 케이스에 대해서는 문제가 생기지 않음을 보증할 수 있고 개발하는 구성원의 두려움도 크게 줄어들 것임

    배운 점

    • 함수(메서드) 라인에 대한 기준을 정립함
      • 함수는 15라인을 넘지 않도록 하며, 이는 main() 함수에도 적용됨. 공백 라인도 한 라인으로 간주됨.
      • 함수 라인을 제한함으로써 코드를 읽기 쉽게 하고, 함수 분리를 통해 유지보수성을 향상시킴.
    • 발생할 수 있는 예외 상황을 고민함
      • 정상적인 경우뿐만 아니라 예외 상황도 고려하여 프로그래밍하는 습관을 들임.
      • 예외 상황을 처리함으로써 프로그램의 안정성과 사용자 경험을 개선함.
      // 예시: 예외 상황 처리
function validateLottoPurchaseAmount(amount) {
    if (amount < 1000) {
        throw new Error("Purchase amount must be at least 1000.");
    }
}

function validateLottoNumbers(numbers) {
    let uniqueNumbers = new Set(numbers);
    if (uniqueNumbers.size !== numbers.length) {
        throw new Error("Duplicate numbers are not allowed.");
    }
    numbers.forEach(number => {
        if (number < 1 || number > 45) {
            throw new Error("Numbers must be between 1 and 45.");
        }
    });
}
    • 비즈니스 로직과 UI 로직을 분리함
      • 비즈니스 로직과 UI 로직을 한 클래스에 담지 않고 분리하여 단일 책임 원칙을 준수함.
      • 비즈니스 로직과 UI 로직의 분리는 코드의 유지보수성을 높이고, 테스트를 용이하게 함.
      // 예시: 비즈니스 로직과 UI 로직 분리
class Lotto {
    constructor(numbers) {
        this.numbers = numbers;
    }

    contains(number) {
        return this.numbers.includes(number);
    }
}

class LottoUI {
    static print(numbers) {
        console.log(numbers.join(', '));
    }
}
    • 현재 객체의 상태를 보기 위한 로그 메시지를 toString()으로 구현함
      • 객체의 상태를 보기 위한 로그 메시지는 toString() 메서드를 통해 구현함.
      • toString() 메서드를 사용하여 객체의 상태를 쉽게 확인하고 디버깅을 용이하게 함.
    • 연관성이 있는 상수는 static final 대신 enum을 활용함
      • 연관성이 있는 상수는 enum을 활용하여 그룹화함.
      • enum을 사용하여 코드의 가독성을 높이고 상수 관리의 일관성을 유지함.
      // 예시: enum 사용
const Rank = Object.freeze({
    FIRST: { matches: 6, prize: 2000000000 },
    SECOND: { matches: 5, prize: 30000000 },
    THIRD: { matches: 5, prize: 1500000 },
    FOURTH: { matches: 4, prize: 50000 },
    FIFTH: { matches: 3, prize: 5000 },
    MISS: { matches: 0, prize: 0 }
});
    • final 키워드를 사용해 값의 변경을 막음
      • 값의 변경을 막기 위해 final 키워드를 사용함.
      • final 키워드를 사용하여 객체의 불변성을 유지하고, 예측 가능한 코드를 작성함.
      // 예시: final 키워드 사용
class Money {
    constructor(amount) {
        this.amount = Object.freeze(amount);
    }
}
    • 객체의 상태 접근을 제한함
      • 인스턴스 변수의 접근 제어자는 private으로 구현하여 객체의 상태 접근을 제한함.
      • 접근 제어자를 사용하여 객체의 캡슐화를 강화하고 데이터 무결성을 유지함.
      // 예시: 접근 제어자 사용
class WinningLotto {
    #lotto;
    #bonusNumber;

    constructor(lotto, bonusNumber) {
        this.#lotto = lotto;
        this.#bonusNumber = bonusNumber;
    }
}
    • 객체를 객체스럽게 사용함
      • 데이터만 가지는 객체에 로직을 추가하여 객체를 객체답게 사용함.
      • 메시지를 던지도록 구조를 변경하여 객체가 스스로 일을 하도록 함.
      // 예시: 객체를 객체스럽게 사용
class Lotto {
    constructor(numbers) {
        this.numbers = numbers;
    }

    contains(number) {
        return this.numbers.includes(number);
    }

    matchCount(otherLotto) {
        return this.numbers.filter(number => otherLotto.contains(number)).length;
    }
}
    • 필드(인스턴스 변수)의 수를 줄이기 위해 노력함
      • 불필요한 필드나 중복되는 필드를 줄여 객체의 복잡도를 낮춤.
      • 필드 수를 최소화하여 코드의 가독성과 유지보수성을 향상시킴.
      // 예시: 필드 수 줄이기
class LottoResult {

    • 성공하는 케이스 뿐만 아니라 예외에 대한 케이스도 테스트함
      • 성공하는 케이스뿐만 아니라 예외 상황에 대한 테스트도 작성함.
      • 입력 데이터의 유효성 검사, 경계 조건 처리, 다중 스레딩 및 동시성, 경계 값 테스트를 통해 결함 발생 가능성을 줄이고 코드의 신뢰성을 높임.
      // 예시: 예외 상황 테스트
describe('Lotto', () => {
    it('throws error if bonus number is duplicate', () => {
        expect(() => {
            new WinningLotto(new Lotto([1, 2, 3, 4, 5, 6]), 6);
        }).toThrow('Bonus number should not be a duplicate of winning numbers');
    });
});
    • 테스트 코드도 코드임을 인식하고 리팩터링함
      • 테스트 코드도 코드이므로 반복되는 부분을 중복되지 않게 리팩터링함.
      • 리팩터링된 테스트 코드는 유지보수성을 높이고 테스트 작성 시간을 줄임.
      • 테스트 코드를 작성할 때, ‘Given-When-Then’ 형식을 활용해서 가독성을 높여보기
      // 리팩터링 전:
describe('Money', () => {
    it('throws error if amount is 999', () => {
        expect(() => new Money(999)).toThrow('Amount must be at least 1000');
    });
  
    it('throws error if amount is 0', () => {
        expect(() => new Money(0)).toThrow('Amount must be at least 1000');
    });
  
    it('throws error if amount is -123', () => {
        expect(() => new Money(-123)).toThrow('Amount must be at least 1000');
    });
});
  
// 리팩터링 후:
describe('Money', () => {
    it.each([999, 0, -123])('throws error if amount is less than 1000', (input) => {
        expect(() => new Money(input)).toThrow('Amount must be at least 1000');
    });
});
    • 테스트를 위한 코드는 구현 코드에서 분리함
      • 테스트를 위한 편의 메서드를 구현 코드에 추가하지 않음.
      • 테스트 전용 메서드를 구현 코드에 추가하지 않아 코드의 순수성과 일관성을 유지함.
    • 단위 테스트하기 어려운 코드를 리팩터링함
      • 테스트하기 어려운 부분은 클래스 외부로 분리하여 단위 테스트가 가능하게 만듦.
      • 테스트하기 쉬운 구조로 리팩터링하여 코드의 테스트 커버리지를 높임.
      // 리팩터링 전:
import camp.nextstep.edu.missionutils.Randoms;
  
class Lotto {
    constructor() {
        this.numbers = Randoms.pickUniqueNumbersInRange(1, 45, 6);
    }
}
  
class LottoMachine {
    execute() {
        const lotto = new Lotto();
    }
}
  
// 리팩터링 후:
class Lotto {
    constructor(numbers) {
        this.numbers = numbers;
    }
}
  
class LottoMachine {
    execute(numberGenerator) {
        const numbers = numberGenerator();
        const lotto = new Lotto(numbers);
    }
}
  
// 테스트 코드
describe('LottoMachine', () => {
    it('should create a lotto with generated numbers', () => {
        const mockNumberGenerator = jest.fn().mockReturnValue([1, 2, 3, 4, 5, 6]);
        const lottoMachine = new LottoMachine();
        lottoMachine.execute(mockNumberGenerator);
        expect(mockNumberGenerator).toHaveBeenCalled();
    });
});
    • private 함수를 테스트하기 위해 클래스(객체) 분리를 고려함
      • 가독성을 위해 분리된 private 함수를 public 함수로 검증 가능하게 함.
      • 필요 시 역할을 분리하여 의미 있는 단위로 객체를 분할하고 테스트를 용이하게 만듦.
        • 함수(function)와 객체(class)를 어떻게 의미 있는 단위로 분할할 지 고민
      // 리팩터링 전:
class Calculator {
    // 공개되지 않은 함수 (#을 이용한 private 지정. ECMAScript 2019(ES10)부터 도입된 기능)
    #add(a, b) {
        return a + b;
    }
  
    // 공개 함수
    calculateSum(a, b) {
        return this.#add(a, b);
    }
}
  
// 테스트 코드
const calculator = new Calculator();
console.log(calculator.calculateSum(2, 3)); // 테스트는 가능하지만, private 함수를 직접 테스트하기 어려움
  
// 리팩터링 후:
class Adder {
    // 공개 함수로 변환
    add(a, b) {
        return a + b;
    }
}
  
class Calculator {
    constructor(adder) {
        this.adder = adder;
    }
  
    calculateSum(a, b) {
        return this.adder.add(a, b);
    }
}
  
// 테스트 코드
const adder = new Adder();
const calculator = new Calculator(adder);
console.log(adder.add(2, 3)); // Adder 클래스의 add 메서드를 직접 테스트 가능
console.log(calculator.calculateSum(2, 3)); // Calculator 클래스의 calculateSum 메서드를 테스트 가능

    마무리

    이번 미션을 통해 세 가지 문제에 직면하게 되었고, 이는 고민해보는 좋은 기회가 되었다.

    첫째, 단위 테스트를 고려한 함수 작성과 테스트 코드 작성에 대한 컨벤션 등 테스트 환경에 대한 경험이 부족해서 많은 시행착오를 겪었다. 함수를 어떻게 작성하면 목적을 명확하게 분리하면서 유지보수 측면에서도 유연하게 할 수 있을지에 대해 깊이 고민했다.

    둘째, 계속해서 코드가 2 depth 이상으로 작성되어 어려움을 겪었다. 특히 이번 미션에서는 에러에 대한 예외처리가 추가되면서 함수를 쪼개도 2 depth 이상으로 구성되는 경우가 빈번했다. 이러한 문제를 어떻게 해결할 수 있을지에 대해 심도 있게 고민했다.

    셋째, Java의 강타입 특성으로 인해 변수명을 할당하고 타입을 바꾸는 과정에서 변수명끼리 의미가 충돌되는 경우가 생겼다. 기존에 정리한 변수명 짓는 규칙이 한계에 부딪힌 상황이었고, 이에 대해 어떻게 개선할 수 있을지 고민했다.

    프리코스를 통해 배운 내용을 고려하면서 작성하려니 아직 서툰 부분이 많았지만, 반복을 통해 체득하게 된다면 이번에 고민했던 문제들을 포함하여 해결할 수 있게 되고 한 단계 도약할 것이라는 생각이 들었다.
