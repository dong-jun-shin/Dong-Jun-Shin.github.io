    과제 진행 셀프 QnA

    • 단위 테스트를 위한 함수 쪼개기
      • indent(depth) 줄이기
      • 3항 연산자 사용하지 않기
    • 커밋 메시지 컨벤션
      • 커밋 메시지 형식: <type>(<scope>): <subject> 형식, 각 줄 100자 이내.
      • Subject line: 간결한 변경 사항 설명, 명령형 현재 시제 사용, 첫 글자 대문자 금지, 마침표 금지
        • 변경 사항의 유형을 지정: 허용되는 유형 feat, fix, docs, style, refactor, test, chore 등
        • 변경 사항의 위치를 지정: 허용되는 범위 $location, $browser 등
      • Message body: 명령형 현재 시제, 변경 이유 및 이전 동작과의 대조 설명
      • Message footer: 호환성 깨짐 설명, 이유, 마이그레이션 방법 명시

    배운 점

    • README.md를 상세히 작성함
      • 프로젝트의 개요와 기능을 마크다운 형식으로 설명하여 다른 사람들이 쉽게 이해하고 활용할 수 있도록 함.
      • 상세한 README.md 작성은 프로젝트의 목적과 기능을 명확히 전달하여 협업과 오픈 소스 기여를 촉진함.
    • 기능 목록을 재검토함
      • 기능 목록은 클래스 설계나 함수 구현보다 상위 수준에서 작성하여 변경 가능성을 염두에 둠.
      • 기능 목록의 재검토는 변경 사항을 유연하게 반영하고 예외 상황을 지속적으로 추가하여 안정성을 높임.
    • 기능 목록을 업데이트함
      • 기능 목록은 구현 과정에서 변경될 수 있으므로 지속적으로 업데이트하여 살아있는 문서를 만듦.
      • 업데이트된 기능 목록은 실제 구현을 반영하여 일관성을 유지하고 개발 프로세스를 명확하게 함.
    • 값을 하드 코딩하지 않음
      • 문자열, 숫자 등의 값을 하드 코딩하지 않고 상수로 정의하여 의도를 명확히 함.
      • 하드 코딩을 피함으로써 코드의 가독성을 높이고 유지보수성을 향상시킴.
       // 예시: 상수 사용
 const MAX_CAR_NAME_LENGTH = 5;
 const MIN_CAR_NAME_LENGTH = 1;

 function validateCarName(name) {
     if (name.length < MIN_CAR_NAME_LENGTH || name.length > MAX_CAR_NAME_LENGTH) {
         throw new Error(`Car name must be between ${MIN_CAR_NAME_LENGTH} and ${MAX_CAR_NAME_LENGTH} characters.`);
     }
 }
    • 구현 순서도 코딩 컨벤션임
      • 클래스는 상수, 멤버 변수, 생성자, 메서드 순으로 작성하여 구조를 일관되게 함.
      • 구현 순서를 유지함으로써 코드의 가독성과 유지보수성을 높임.
       class Car {
     static MAX_SPEED = 200; // 상수
     speed = 0; // 인스턴스 변수

     constructor(name) { // 생성자
         this.name = name;
     }

     accelerate() { // 메서드
         this.speed += 10;
         if (this.speed > Car.MAX_SPEED) {
             this.speed = Car.MAX_SPEED;
         }
     }
 }
    • 변수 이름에 자료형을 사용하지 않음
      • 변수 이름에 자료형이나 자료 구조를 사용하지 않고, 변수의 역할을 명확히 드러내는 이름을 사용함.
      • 의미 있는 변수 이름은 코드의 가독성을 높이고 유지보수성을 향상시킴.
       // 예시: 의미 있는 변수 이름 사용
 let carNames = getCarNames();
 let nameArray = carNames.split(",");
    • 한 함수가 한 가지 기능만 담당하게 함
      • 함수는 한 가지 기능만 수행하도록 하여 길어지면 적절히 분리함.
      • 단일 기능 함수는 코드의 재사용성을 높이고 테스트와 디버깅을 용이하게 함.
       function getUserInput(prompt) {
     console.log(prompt);
     return promptUser();
 }

 function validateInput(input) {
     if (input.length < 1 || input.length > 5) {
         throw new Error("Input must be between 1 and 5 characters.");
     }
 }

 function processInput(input) {
     let inputs = input.split(",");
     inputs.forEach(validateInput);
     return inputs;
 }
    • 함수가 한 가지 기능을 하는지 확인하는 기준을 세움
      • 여러 함수에서 중복되는 코드를 함수로 분리하고, 함수 길이를 15라인 이하로 유지함.
      • 단일 기능 함수는 코드의 가독성과 유지보수성을 높이고 재사용성을 강화함.
    • 테스트를 작성하는 이유에 대해 본인의 경험을 토대로 정리함
      • 테스트는 코드의 기능을 점검하고 빠른 피드백을 제공하며, 학습 도구로도 활용할 수 있음.
      • 테스트 작성 경험을 통해 코드의 안정성과 신뢰성을 높이고 개발 속도를 향상시킴.
    • 처음부터 큰 단위의 테스트를 만들지 않음
      • 작은 단위의 테스트부터 시작하여 코드에 대한 빠른 피드백을 받음.
      • 작은 단위 테스트는 코드의 핵심 기능을 검증하고 큰 단위 테스트보다 빠르게 피드백을 제공함.
        // 예시: 작은 단위 테스트
  function testCarMovement() {
      let car = new Car("Test");
      car.accelerate();
      console.assert(car.speed > 0, "Car should move forward when accelerated.");
  }

  testCarMovement();

    마무리

    작업 전 기능 목록을 정리하고 이를 토대로 함수를 작게 쪼개는 일이 생각보다 어려웠다. 이번 주에는 이 과정에서 두 가지 중요한 점을 배웠다.

    첫째, 기능 목록과 단위 테스트가 잘 일치해야 한다는 점이었다. 기능 목록이 잘 정리되어 있으면 단위 테스트를 작성할 때 어떤 것들을 테스트해야 하는지 쉽게 알 수 있었다. 이는 테스트의 명확성과 효율성을 높이는 데 큰 도움이 되었다.

    둘째, 기능 목록과 커밋 메시지 컨벤션이 잘 일치해야 한다는 점이었다. 기능 목록에 따라 기능 단위로 커밋을 할 때, 코드의 양과 커밋 메시지의 양이 들쭉날쭉한 것을 보고 정리된 기능들의 단위 크기를 시각적으로 확인할 수 있었다. 이는 코드 관리와 팀 협업에 있어 중요한 인사이트를 제공했다.

    이번 미션을 통해 테스트와 커밋 메시지 컨벤션에 대해 많이 고민해보고 경험할 수 있어서 좋았다. 그리고 이전에는 별개의 작업이라고 생각했던 기능 목록 작성, 코드 작성, 테스트, 커밋 메시지 컨벤션이 하나의 흐름으로 연결된다는 것을 깨달았다. 이 경험은 앞으로의 개발 작업에 큰 도움이 될 것 같다.
