    CSR과 SSR, 백엔드와 프론트엔드의 개발영역

    CSR과 SSR

    • CSR은 ‘Client-Side Rendering’이라는 뜻으로 클라이언트에서 페이지를 구성해서 보여주는 방식을 말한다.
    • SSR은 ‘Server-Side Rendering’이라는 뜻으로 서버에서 페이지를 구성해서 보여주는 방식을 말한다. JSP/Servlet의 아키텍처에서 이 방식을 사용했다.

    CSR 특징

    • 장점
      • 콘텐츠 구성이 시작되는 초기 전송 페이지의 속도가 빠르다
    • 단점
      • 서비스에 필요한 데이터를 클라이언트(브라우저)에서 추가로 요청하고 재구성해야 하기 때문에 구성이 완료된 콘텐츠를 보여주는 시점은 SSR보다 느리다.

    SSR 특징

    • 장점
      • 구성이 완료된 콘텐츠를 사용자에게 보여주는데 걸리는 시간이 CSR보다 빠르다.
      • 개발 영역의 완전한 분리로 Front와 Back 사이에 API 명세에 대한 소통으로 국한되어 커뮤니케이션 비용 절감
      • Front의 작업 완성도 및 이슈 대응 생산성 향상
    • 단점
      • 콘텐츠를 구성하는 속도는 CSR보다 느리다.

    CSR 사례

    1. Angular.js
      • CSR만을 지원하는 프레임워크 (SSR을 위한 공식 지원은 X)
        • SPA(single-page application)를 구축하는데 유용
        • 사용자 UI의 요구사항이 많아서, 빠르고 기민한 대응을 위한 프레임워크가 필요할 때 사용
        • 페이지 로드 후, 동적으로 콘텐츠를 생성
        • UI가 아닌 콘텐츠를 빠르게 소비하는 사용자의 요구 사항을 충족시키기에는 부족
    2. SPA(single-page application)
      • 단일 페이지 애플리케이션
      • 현재 페이지를 동적으로 작성해서 사용자와 소통하는 웹 애플리케이션
      • 연속되는 페이지 간의 사용자 경험을 향상
      • 데스크톱 애플리케이션처럼 동작
      • Frontend 영역과 Backend 영역
      • CSR View, SSR View | API로 분리
      • 기존 개발
      • CSR(SPA) 개발

    SSR 사례

    1. Node.js
      • SSR을 구축할 때, React의 기반으로 JavaScript를 최대한 활용과 React 생태계 이용이 가능
      • 개발의 난이도가 있지만, 생산성 측면에서 우수
      • Front와 Back을 완전 분리 후, REST API로 연결
      • 가상화를 통한 인프라 구축과 운영
      • Frontend 영역과 Backend 영역
        • 페이지 개발 | API로 분리
        • CSR(SPA) 개발
        • SSR 개발

    CSR과 SSR을 선택할 때 고려할 점

    • 초기 비용, 요구사항 등 설계 환경과 유지보수 비용 등 운영 환경을 고려해서 선택
    • 필요에 따라 CSR, SSR의 비중을 조정해서 구축

    주제 이외 개념

    1. Angular.js : SPA 개발을 편리하게 하는 프론트엔드 프레임워크
    2. Node.js : 자바스크립트 런타임 환경
    3. React : Dom을 조작할 수 있는 프론트엔드 프레임워크
